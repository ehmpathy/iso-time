# vision: true UTC timezone handling

## the outcome world

### before

a developer in new york (UTC-5) runs:
```typescript
import { now, asIsoTimeStamp } from 'iso-time';

const timestamp = now(); // at 9:00 AM local time
console.log(timestamp);  // '2026-03-30T09:00:00Z' ← WRONG: local time with fake Z
```

the timestamp claims to be UTC (the `Z` suffix) but is actually local time. downstream systems that trust the `Z` suffix will be off by hours. tokens expire early. scheduled jobs fire at wrong times. audit logs are wrong.

### after

```typescript
import { now, asIsoTimeStamp } from 'iso-time';

const timestamp = now(); // at 9:00 AM local time (14:00 UTC)
console.log(timestamp);  // '2026-03-30T14:00:00Z' ← CORRECT: actual UTC
```

the timestamp is always correct UTC. tokens expire when expected. jobs fire on time. audit logs are accurate regardless of where the code runs.

### the "aha" moment

a developer who debugs why their keyrack keys "expire immediately" discovers the bug: the 30-minute TTL they set is calculated from fake-UTC, which is actually 4.5 hours in the past. the keys are born expired.

after the fix: they set 30 minutes, they get 30 minutes. no surprises.

## user experience

### usecases

| goal | contract | behavior |
|------|----------|----------|
| get current time | `now()` | returns actual UTC timestamp |
| cast to timestamp | `asIsoTimeStamp(date)` | converts to actual UTC |
| get today's date | `today()` | returns local date (intentional) |
| add duration | `addDuration(stamp, dur)` | arithmetic stays in UTC |

### timeline

1. user imports `iso-time`
2. user calls `now()` or `asIsoTimeStamp(new Date())`
3. user gets correct UTC timestamp regardless of their timezone
4. user stores/transmits timestamp with confidence
5. downstream systems parse timestamp correctly

### what it looks like

```typescript
// tokyo (UTC+9): local time is 2026-03-30 23:00
// new york (UTC-5): local time is 2026-03-30 09:00
// both get the same UTC time:

import { now } from 'iso-time';
console.log(now()); // '2026-03-30T14:00:00Z' ← same everywhere
```

## mental model

### how users describe it

> "iso-time gives me UTC timestamps. doesn't matter where my server runs."

> "it's like `new Date().toISOString()` but typed and without milliseconds."

### analogies

- **GPS coordinates**: just like lat/lng are absolute regardless of where you measure them, iso-time timestamps are absolute UTC regardless of where the code runs
- **unix epoch**: milliseconds since 1970 don't care about timezones; iso-time is the human-readable version of that

### terms

| user says | we say |
|-----------|--------|
| "the time" | UTC timestamp |
| "local time" | (not supported — use date-fns directly) |
| "timezone" | (irrelevant — we only speak UTC) |

## evaluation

### how well does it solve the goals?

| goal | status |
|------|--------|
| correct UTC output | ✅ native `.toISOString()` is always correct |
| no new dependencies | ✅ uses built-in Date methods |
| backwards compatible format | ✅ same `yyyy-MM-ddTHH:mm:ssZ` format |
| type safety preserved | ✅ branded types unchanged |

### pros

- **always correct**: native JS `.toISOString()` is timezone-proof
- **no dependencies**: removes reliance on `date-fns/format` for UTC conversion
- **testable**: can run tests in any timezone and they pass
- **simple**: fewer parts that move = fewer bugs

### cons

- **compat-break for wrong usage**: code that depended on the buggy behavior will break (but that code was already wrong)

### edgecases & pit of success

| edgecase | handled by |
|----------|------------|
| midnight boundary | `today()` returns local date; `now()` returns UTC timestamp |
| DST transition | irrelevant — UTC has no DST |
| leap seconds | JS Date (ignored like everywhere) |
| pre-epoch dates | works (negative timestamps) |

the fix keeps users in a pit of success by the simple path (`now()`, `asIsoTimeStamp()`) always correct. there's no way to accidentally get local time when you wanted UTC.

## open questions & assumptions

### assumptions

1. **users want UTC**: the `Z` suffix in `IsoTimeStamp` means UTC, so users expect UTC
2. **local time is not needed**: if users need local time, they should use `date-fns` directly
3. **format stays the same**: `yyyy-MM-ddTHH:mm:ssZ` (no milliseconds)

### questions

1. **[answered] IsoDateStamp and IsoMonthStamp scope**: these also use `date-fns/format()` and give local date/month. should we fix those too?
   - **answer**: no — dates are intentionally local timezone. only timestamps (`IsoTimeStamp`) with `Z` suffix claim to be UTC and need the fix. dates have no timezone indicator, so local is valid. docs updated to clarify.

## what is awkward?

### date vs timestamp semantics

`IsoDateStamp` (`2026-03-30`) has no timezone indicator. the decision:
- **timestamps** (`IsoTimeStamp`): always UTC (has `Z` suffix)
- **dates** (`IsoDateStamp`, `IsoMonthStamp`): local timezone (no timezone indicator)

**mitigation**: docs updated to clarify this distinction.

### break in compat

code that accidentally worked because CI runs in UTC will break when run locally in other timezones. this is actually good — it surfaces latent bugs — but may cause initial confusion.

**mitigation**: clear changelog that notes the behavior change.

### format truncation

native `.toISOString()` returns `2026-03-30T14:00:00.000Z` (with milliseconds), but `IsoTimeStamp` format is `2026-03-30T14:00:00Z` (without). we need to truncate.

```typescript
date.toISOString().replace(/\.\d{3}Z$/, 'Z')
```

this is safe and deterministic.
